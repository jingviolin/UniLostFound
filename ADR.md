# Architecture Decision Records (ADR) - UniLostFound

本文件记录了项目开发过程中的关键架构决策、权衡及其背景。

## ADR 1: 采用 Supabase 作为后端即服务 (BaaS)
- **状态**：已通过 (Accepted)
- **日期**：2026-05-18 (Week 6)
- **上下文**：项目需要快速实现数据持久化、身份验证和实时功能，且开发周期较短（零代码选修课）。
- **决策**：选择 Supabase 而非自建 Node.js + MongoDB/PostgreSQL。
- **原因**：
    1.  **内置 Auth**：无需编写注册登录逻辑，直接使用现成的 API。
    2.  **RLS (行级安全)**：直接在数据库层解决隐私隔离问题（私聊仅参与者可见）。
    3.  **零部署成本**：无需管理服务器。
- **后果**：
    - 优点：极大提升开发速度。
    - 缺点：业务逻辑深度绑定 Supabase，未来迁移成本高。

## ADR 2: 采用 DeepSeek-V3 配合 JSON Mode
- **状态**：已通过 (Accepted)
- **日期**：2026-05-20 (Week 6)
- **上下文**：需要从用户输入的非结构化文本中提取结构化字段（物品名、地点等）。
- **决策**：使用 DeepSeek API 并开启 `response_format: { type: 'json_object' }`。
- **原因**：
    1.  **性价比**：相比 GPT-4o 成本极低，性能足以应对简单的信息提取。
    2.  **稳定性**：JSON Mode 确保了前端解析的鲁棒性。
- **后果**：
    - 优点：前端可以直接 `JSON.parse`，无需正则匹配。

## ADR 3: 引入 Human-in-the-loop (HITL) 智能发布流水线
- **状态**：已通过 (Accepted)
- **日期**：2026-05-25 (Week 7)
- **上下文**：AI 提取可能存在幻觉（如编造不存在的地点）。
- **决策**：在 AI 解析和最终入库之间增加一个“人工核对”环节。
- **原因**：
    - **系统可靠性**：用户作为最后一道防线，确保入库数据的准确性。
    - **Agentic Workflow**：将任务拆解为五个节点，使流程可解释、可干预。
- **后果**：
    - 优点：提升了数据质量，减少了无效匹配。
    - 缺点：增加了一次点击操作，牺牲了部分极致的“一键发布”体验。

## ADR 4: 实现系统免疫系统 (Observability & Reliability)
- **状态**：已通过 (Accepted)
- **日期**：2026-05-25 (Week 8)
- **上下文**：真实环境下 API 可能超时，用户不知道系统后台发生了什么。
- **决策**：
    1.  **指数退避重试**：针对 API 调用实现自动重试逻辑。
    2.  **结构化日志**：记录每一个节点的输入、输出和耗时。
    3.  **日志面板**：在前端展示简易日志，方便调试。
- **原因**：
    - **工程韧性**：确保系统在异常情况下不崩溃。
